## 1. Sources reviewed (retrieval-dated 2026-05-14)

| #   | Topic                                                   | URL                                                                                  | Notes                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| --- | ------------------------------------------------------- | ------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | TypeScript 6.0 — `moduleResolution: node10` deprecation | <https://github.com/microsoft/TypeScript/issues/62200>                               | `node10` issues an error in TS 6 unless `--ignoreDeprecations` is set; removed entirely in TS 7. The reason: `node10` predates `package.json` `exports`/`imports`, CJS/ESM file distinction, and explicit-extension rules — leads to widespread misconfiguration.                                                                                                                                                                                                                                       |
| 2   | TypeScript 6.0 — recommended replacement                | <https://github.com/microsoft/TypeScript/pull/62338>                                 | For Node.js apps: switch to `--module nodenext` (implies `--moduleResolution node`). For bundler apps: `--module esnext` + `--moduleResolution bundler`. `node16` + `nodenext` remain supported for modern Node.                                                                                                                                                                                                                                                                                        |
| 3   | TS `moduleResolution` reference                         | <https://www.typescriptlang.org/tsconfig/moduleResolution.html>                      | `Node16` = pinned to Node 16 module-resolution rules; `NodeNext` = tracks current Node. For Nest 11 (Node 20+ floor), both are equivalent today; `Node16` is the stabler pin.                                                                                                                                                                                                                                                                                                                           |
| 4   | TypeScript 6.0 release notes                            | <https://devblogs.microsoft.com/typescript/announcing-typescript-6-0/>               | Major version released 2026 (per the issue's iteration plan); core change is the `node10` deprecation. `baseUrl` deprecation noted separately — paths can still work without it when relative resolution suffices.                                                                                                                                                                                                                                                                                      |
| 5   | ESLint 9 flat-config migration                          | <https://eslint.org/docs/latest/use/migrate-to-9.0.0>                                | `.eslintrc*` formats removed; `eslint.config.{js,mjs,ts}` mandatory. v9.x EOL 2026-08-06 — we'll bump to v10 in a follow-up before then.                                                                                                                                                                                                                                                                                                                                                                |
| 6   | ESLint flat-config `extends` + `defineConfig`           | <https://eslint.org/blog/2025/03/flat-config-extends-define-config-global-ignores/>  | `defineConfig()` is the new recommended factory (March 2025). Supports inheritance via `extends` again (the original biggest friction in v9 migration). Available from `eslint/config` package.                                                                                                                                                                                                                                                                                                         |
| 7   | typescript-eslint shared configs                        | <https://typescript-eslint.io/users/configs>                                         | Recommended for typed lint: `recommended-type-checked` + `stylistic-type-checked` + `projectService: true` for perf. Per-package opt-in is the pattern we follow.                                                                                                                                                                                                                                                                                                                                       |
| 8   | Prettier 3.x (lockfile resolves to 3.6.2)               | <https://prettier.io/versions.html>                                                  | The latest published `3.x` line as of 2026-05-14 is 3.8.1, but our root `prettier` devDep specifier is `^3.2.5` and the lockfile resolves to 3.6.2. PR-9 doesn't bump the dep on purpose (separate concern from polish); a follow-up will land the bump alongside `eslint-plugin-prettier` if/when we add it. Defaults at 3.6.2: `printWidth=80`, `tabWidth=2`, `useTabs=false`, `semi=true`, `singleQuote=false`, `trailingComma=all`, `arrowParens=always`. Our only override is `singleQuote: true`. |
| 9   | Prettier configuration reference                        | <https://prettier.io/docs/configuration/>                                            | Supports `.prettierrc.{js,mjs,cjs,ts,json,yml}` + `prettier` key in package.json + `prettier.config.js`. We use `.prettierrc.mjs` (ESM-native, matches workspace `type: "module"`).                                                                                                                                                                                                                                                                                                                     |
| 10  | Jest 30 configuration                                   | <https://jestjs.io/docs/30.0/configuration>                                          | Supports `coverageThreshold` (global + per-file), `preset: 'ts-jest'`, `testEnvironment`, `transform`. We use the modular config shape (`base.ts` → `nest.ts`/`next.ts`/`library.ts`) so each consumer extends just what it needs.                                                                                                                                                                                                                                                                      |
| 11  | Jest 30 upgrade notes                                   | <https://jestjs.io/docs/30.0/upgrading-to-jest30>                                    | Node 18+ floor; TS 5.4+ floor. Deprecated matcher aliases removed (`toBeCalled()` → `toHaveBeenCalled()`). All our existing tests already use the modern matchers.                                                                                                                                                                                                                                                                                                                                      |
| 12  | NestJS 11 release notes                                 | <https://github.com/nestjs/nest/releases/tag/v11.0.0>                                | Released Jan 2025. Node 20+ floor (v16/v18 dropped). Nest core remains CJS-based; `Node16`/`NodeNext` `moduleResolution` is safe and required for consuming ESM-only deps.                                                                                                                                                                                                                                                                                                                              |
| 13  | NestJS + NodeNext relative imports                      | <https://stackoverflow.com/questions/74830166/unable-to-import-esm-module-in-nestjs> | Under `moduleResolution: "Node16"`/`"NodeNext"`, relative imports MUST carry the `.js` extension (even though the source file is `.ts`). This is the cascade PR-9 applies to `apps/api-gateway/src/`.                                                                                                                                                                                                                                                                                                   |
| 14  | `prettier` package                                      | <https://www.npmjs.com/package/prettier/v/3.3.0>                                     | Treated as peer-dep in our consuming packages so each can pin its own version; lockfile resolves to 3.8.1.                                                                                                                                                                                                                                                                                                                                                                                              |

## 2. Decisions log

| #   | Decision                               | Choice                                                                                                                   | Rejected                                                                                                  | Reasoning                                                                                                                                                                              |
| --- | -------------------------------------- | ------------------------------------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| D1  | TS 6 deprecation handling              | Modernize: drop `Node10` + `baseUrl` (per user Q2-decision)                                                              | `ignoreDeprecations: "5.0"` shim                                                                          | Aligns with TS 6's recommended path (source #1, #2); pays the migration cost once now while only `apps/api-gateway/src/*` needs the `.js` cascade.                                     |
| D2  | nestjs.json `moduleResolution`         | `Node16` (stabler pin; same behavior as `NodeNext` for Node 20+)                                                         | `NodeNext` (tracks future Node — unnecessary churn for our floor), `Bundler` (Nest doesn't use a bundler) | Nest 11 supports `Node16` cleanly (source #12); `Node16` won't drift if a future Node release changes resolution semantics.                                                            |
| D3  | nestjs.json `baseUrl` removal          | Drop entirely; descendant configs (`apps/api-gateway/tsconfig.json`) drop too                                            | Keep `baseUrl: "./"` with explicit value                                                                  | `baseUrl` is deprecated in TS 6; nothing in our codebase uses path-relative-to-baseUrl imports yet — clean removal.                                                                    |
| D4  | api-gateway import cascade scope       | Relative imports only (`./foo` → `./foo.js`); workspace imports (`@repo/api`) stay extension-less (resolve through pnpm) | Convert workspace to relative aliases too                                                                 | Q5 from sub-plan. Workspace imports are resolved by pnpm's link, not Node's module resolution; no `.js` required.                                                                      |
| D5  | `@repo/prettier-config` shape          | Single `src/index.ts` exporting a `Config` const with project standards                                                  | Multi-file split                                                                                          | One config object — splitting adds indirection without value.                                                                                                                          |
| D6  | prettier-config location               | `@repo/prettier-config` (canonical home)                                                                                 | Stay in `@repo/eslint-config/prettier-base`                                                               | Avoids cross-package coupling; `@repo/eslint-config` should be exclusively ESLint surface.                                                                                             |
| D7  | `@repo/eslint-config/library.js` shape | Pure flat-config matching `base.js` pattern                                                                              | Keep mixed legacy format                                                                                  | The current file mixes flat-config + `.eslintrc`-style `env`/`settings` — the legacy keys are ignored under v9 flat-config; rewriting removes confusion.                               |
| D8  | `@repo/jest-config` library config     | New `src/library.ts` with ts-jest + jsdom-when-needed + coverage thresholds (commented-out default)                      | Reuse `nest.ts` for non-Nest packages                                                                     | Semantic mismatch; library packages don't need `globals.jest` + Nest-specific globals.                                                                                                 |
| D9  | Coverage thresholds shipping           | Ship the threshold objects in code but COMMENTED-OUT by default; each consumer un-comments when they have real tests     | Enforce 80% floor from day 1 across all configs                                                           | Q4. Most packages are pre-test; ship-and-enforce immediately would CI-red the workspace. Opt-in per package.                                                                           |
| D10 | Phase 1 issues lifecycle               | Create at execution start; close at PR-9 merge (Phase 1 is single-PR per parent plan)                                    | Stay open for follow-up phase-1-adjacent work                                                             | Q2. Future P1-adjacent fixes (eslint-missing-in-services, scaffold-package.ts enhancements) get fresh issues with `phase/1` label rather than re-opening the Acceptance issue.         |
| D11 | TS `module` field                      | Defer changing `module` to `Node16` until P4 rewrites api-gateway src                                                    | Update `module` now alongside `moduleResolution`                                                          | Q1. Changing `module` to `Node16` forces ESM emit semantics on Nest, which currently runs as CJS. `moduleResolution: Node16` works regardless of the `module` value; safer split.      |
| D12 | `apps/api-gateway/tsconfig.json` shape | Override `moduleResolution: "Node16"` directly on the consumer (not in shared `nestjs.json`)                             | Make `nestjs.json` the carrier                                                                            | Cleaner: shared `nestjs.json` is the recommended baseline; each consumer can override per its Node version. Mirrors how `react-library.json` lets consumers add their own `jsx` value. |
